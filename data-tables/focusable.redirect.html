Footnotes

  • The DOM Element Interface is missing the focus method.
  • The focus event is not emitted upon the element becoming the active element (document.activeElement).

  • Note 1

    Considering the <html> and <body> element focusable should have no benefit to a website (unless explicitly made focusable by adding the tabindex attribute), which is why ally.js considers the <html> and <body> element inert.

  • Note 2

    The <html> element itself is not considered focusable, but some browsers may give it focus when focus is passed from browser UI to the document.

  • Note 3

    The <body> element itself is not considered focusable, but it is has focus (i.e. is document.activeElement) if no other element has focus.

  • Note 4

    should not be focusable as per disabled elements. Blink 453847, Webkit 141086

  • Note 5

    The activation behavior of the <label> element is not defined beyond »… should match the platform's label behavior.«

  • Note 6

    The CSS property user-modify was proposed and dropped from CSS UI Level 3 and has thus not become a standard yet.

  • Note 7

    The value is invalid according to rules for parsing integers required by HTML5 tabindex

  • Note 8

    tabindex="" is parsed to the value -32768. Trident 1072965

  • Note 9

    The value's trailing spaces are considered invalid according to rules for parsing integers required by HTML5 tabindex

  • Note 10

    The value's trailing non-numeric characters are considered invalid according to rules for parsing integers required by HTML5 tabindex

  • Note 11

    The <map>'s <area>s are inert (not focusable) as long as the image is still loading.

  • Note 12

    In some browsers adding a tabindex attribute to an <area> element will make it inert (not focusable).

  • Note 13

    In some browsers <img> elements with an associated image map cannot be made focusable.

  • Note 14

    In some browsers the image map associated with an <img> element can be removed from the document's sequential navigation order by adding tabindex="-1" to the <img> element.

  • Note 15

    Some browsers will not make an image map focusable, if it is associated with an <img> that does have a proper image loaded.

  • Note 16

    <audio> is considered interactive content only with the <code>controls</code> attribute present

  • Note 17

    <video> is considered interactive content only with the <code>controls</code> attribute present

  • Note 18

    Firefox does not dispatch focus event but document.activeElement is updated properly. Gecko 131784

  • Note 19

    Microsoft Edge requires the <iframe>d document to contain a focusable element in order to make it keyboard accessible.

    This is undetectable because we can't look inside all iframes, because of the Same Origin Policy.

  • Note 20

    Because of the Same Origin Policy we can't reliably look inside <iframe>s to determine their content, which is why ally.js considers all <iframe>s keyboard focusable (tabbable).

  • Note 21

    Technically the content of an iframe can be accessible to script.

    However, this test is only verifying keyboard focusability.

  • Note 22

    Because of the Same Origin Policy we can't reliably look inside <iframe>s to determine their content, which is why ally.js considers all <iframe> content inert

  • Note 23

    The behavior of the <embed> element depends on the content type and browser plugin.

  • Note 24

    The <video> element should be used for embedding video content instead of <embed> element. The HTML5 <video> element is supported virtually everywhere.

  • Note 25

    The behavior of the <object> element depends on the content it presents. The most common content types used with <object> are SVG and SWF.

  • Note 26

    This element is actually tabbable (keyboard focusable).

    The test shows focusable because the elements were removed from the tabbing test to avoid IE10 from getting focus stuck in the browser UI. This seems to be a problem similar to Trident 1109008.

  • Note 27

    SVG 1.1 does not specify much in respect to accessibility. SVG 2 will bring the tabindex attribute. SVG Tiny 1.2 knows the focusable attribute.

  • Note 28

    By registering a focus event listener the element becomes focusable. Blink 445798, WebKit 140024.

    This is undetectable because elements don't provide a list of their registered event handlers.

  • Note 29

    Elements made focusable by way of adding a focus event listener cannot be identified by ally.js

  • Note 30

    IE10 does not support the hidden attribute

  • Note 31

    The <details> element is specified in HTML 5.1 and not implemented everywhere.

  • Note 32

    Internet Explorer turns <div> and <span> elements focusable when they're scrollable, but does not do the same for other sectioning or block-level elements

  • Note 33

    The ismap attribute makes the <img> focusable (in addition to the parent <a>)

  • Note 34

    There is no indication that the focusability of an element can be inherited by its children, let alone triggered by flexbox layout.

  • Note 35

    naturally focusable according to HTML5 tabindex

  • Note 36

    Firefox transforms <keygen> to <select _moz-type="-mozilla-keygen"> while parsing HTML.

  • Note 37

    The <keygen> element is poorly supported, practically never used and has seen intent to deprecate

  • Note 38

    The <keygen> element is ignored by ally.js
